# Architect-view: История 1 (техническая оценка)

## Контекст истории
**История 1:** владелец добавляет питомца и получает персональный план прививок с датами, чтобы вести вакцинацию в одном месте и не зависеть от конкретной ветклиники.

## Что нужно реализовать (минимально)
1. **Создание сущности `Pet`**: идентификатор, тип (кот/собака), дата рождения (или другой входной параметр для расчета), базовые данные для отображения.
2. **Источник “расписания вакцинаций”**: набор правил/шаблонов обязательных прививок (в MVP — фиксированный набор).
3. **Генерация персонального плана**: расчет дат вакцинаций относительно даты рождения питомца по выбранным шаблонам.
4. **Хранение и показ плана**: список предстоящих прививок с датами (и статусом “запланировано”).

## Предложенная архитектура для MVP (KISS / YAGNI)
### 1) Доменная модель
- `Pet`
  - `id`
  - `animalType`: `cat` | `dog`
  - `birthDate` (или иной обязательный ввод)
- `VaccineTemplate`
  - `animalType`
  - `code`/`name`
  - `offset` (например, “в 8 недель”, “в 12 недель” — через смещение от `birthDate`)
  - `isMandatory` (для фильтра)
- `PlannedVaccine`
  - `petId`
  - `vaccineTemplateId`
  - `scheduledDate`
  - `status`: planned/performed

### 2) Поток (основной happy path)
1. Пользователь добавляет питомца.
2. Система выбирает `VaccineTemplate` по `animalType` и фильтру `isMandatory`.
3. Система вычисляет `scheduledDate = birthDate + offset`.
4. Сохраняет `PlannedVaccine` и показывает план.

## Оценка сложности
- **Расчет дат по смещениям**: низкая/средняя (основная сложность обычно в правилах смещений и единицах времени).
- **Хранение/отображение плана**: низкая.
- **Согласование медицинских схем**: средняя, риск выше, чем “техническая сложность”.
- **Поддержка “всё и сразу” (коты + собаки + кастомные прививки + клинические отличия)**: средняя/высокая из-за расширения модели правил и увеличения количества граничных случаев.

## Технические риски и как их снизить
1. **Неточность медицинского расписания**
   - Риск: неверные смещения или неполное покрытие обязательных прививок.
   - Снижение: начать с узкого MVP (только обязательные прививки для одного вида), затем расширять.
2. **Неопределенный ввод даты рождения**
   - Риск: пользователь не знает дату/она “примерная”.
   - Снижение: MVP потребует точную дату; дальнейшие улучшения — позже.
3. **Часовые пояса и формат дат**
   - Риск: сдвиг даты на сутки при хранении/вычислении.
   - Снижение: хранить даты в UTC или в формате “дата без времени”, вычисления делать в “date-only” семантике.
4. **Эксплуатация правил/шаблонов**
   - Риск: код правил становится неуправляемым при росте.
   - Снижение: вынести шаблоны в конфигурацию/таблицы, чтобы менять набор без переписывания логики.

## Рекомендации по разбиению истории 1 (чтобы было ценностно и управляемо)
Чтобы история давала пользу “сразу” и минимизировала риски, разумный путь:
1. **Базовый график для котов (обязательные прививки)**  
   Пользователь добавляет кота с датой рождения и получает только обязательный план прививок с датами.
2. **Расширение на собак (обязательные прививки)**  
   Аналогично, добавление собаки -> план обязательных прививок.
3. **Добавление кастомных прививок/расширение шаблонов**  
   Только после того, как базовый сценарий устойчив.
4. **Интеграции/выгрузки (например, в календарь)**  
   Как следующий шаг, чтобы не раздувать MVP и не добавлять зависимостей раньше времени.
