# Example Mapping: Персональный план обязательных прививок для кота

## История
Как владелец кота  
Я хочу добавить кота (дата рождения) и получить план обязательных прививок с датами  
Чтобы я мог(ла) вести вакцинацию в одном месте и не пропускать сроки.

## Правила
1. Кот добавлен с обязательным `birthDate` (дата рождения)
   - Пример A: `birthDate = 2026-03-01` -> создается профиль кота и запускается генерация плана
   - Пример B: `birthDate` не указан -> система не генерирует план и сообщает об ошибке/недостатке данных
   - Пример C: `birthDate` указана как дата (без времени) -> план рассчитывается по “date-only” логике

2. Для кота генерируется план **только обязательных** прививок (MVP-фокус)
   - Пример A: в наборе шаблонов есть обязательные и необязательные -> в плане отображаются только обязательные прививки
   - Пример B: пользователь/система еще не поддерживает кастомные прививки -> кастомные прививки не добавляются в план

3. Даты прививок вычисляются как `birthDate + offset` из соответствующего обязательного шаблона
   - Пример A: для шаблона с оффсетом “в 8 недель” -> `scheduledDate = birthDate + 8 недель`
   - Пример B: для шаблона с оффсетом “в 12 недель” -> `scheduledDate = birthDate + 12 недель`
   - Пример C: если `birthDate` попадает в високосный год (например, `2024-02-29`) -> система корректно вычисляет даты, не сдвигая день из‑за логики времени

4. При создании плана каждая запланированная прививка имеет статус `planned`
   - Пример A: сразу после генерации -> все элементы плана имеют статус “запланировано”

## Вопросы
1. Какие именно “обязательные” прививки входят в MVP для котов и какие у них точные оффсеты (например, “в 8 недель”, “в 12 недель” — это пример или нужно зафиксировать реальные значения)?
2. План должен показывать только **будущие** прививки относительно текущей даты или также “в прошлом, но еще актуальные” (если пользователь добавил кота позже)?
3. В какой формат и единицы округления приведены даты при вычислении (например, если оффсет приводит к дате “ровно в полдень” на стороне клиента)?
4. Как обрабатываем ситуацию, когда пользователь знает дату рождения “примерно” (в MVP это запрещено, но нужно зафиксировать правило: ошибка или аппроксимация)?

## Готовность
- Готово к разработке: нет (есть вопросы по составу/оффсетам обязательных прививок и по правилам “будущее/все даты”).
- Что нужно уточнить: пункты в разделе “Вопросы”.
- Следующее действие: зафиксировать состав обязательных шаблонов для котов и правила показа плана относительно текущей даты.

# Example Mapping: Список ближайших прививок и отметка статуса

## История
Как владелец кота или собаки  
Я хочу видеть список ближайших запланированных прививок и отмечать выполненные  
Чтобы не пропускать даты и понимать актуальный статус вакцинации для каждой прививки.

## Правила
1. Система показывает пользователю список запланированных прививок для соответствующего питомца (контекст может быть “только один питомец” в MVP)
   - Пример A: в аккаунте/в MVP один питомец -> показывается список прививок этого питомца
   - Пример B: в списке нет запланированных прививок -> отображается пустое состояние (нет данных)

2. “Ближайшие” выбираются по `scheduledDate` по возрастанию и ограничиваются верхним пределом (предложение для MVP)
   - Пример A: ближайшие = следующие 5 по `scheduledDate` -> выводится ровно 5 ближайших (если их меньше — выводится меньше)
   - Пример B: среди ближайших есть прививка на сегодня -> она попадает в список и имеет `scheduledDate` без сдвига

3. На карточке/в строке каждой прививки показывается `scheduledDate` и текущий статус
   - Пример A: статус `planned` -> отображается как запланировано
   - Пример B: статус `performed` -> отображается как выполнено (если UI показывает историю)

4. Пользователь может отметить прививку как выполненную: статус меняется `planned -> performed`
   - Пример A: пользователь отмечает запланированную прививку -> статус становится `performed` и сохраняется
   - Пример B: пользователь повторно отмечает уже выполненную прививку -> статус не меняется (идемпотентность), дубликаты не создаются

5. Пересчёт дат следующих прививок при отметке “выполнено” не выполняется (MVP-фокус: статусы обновляются, расписание основано на дате рождения)
   - Пример A: после отметки “выполнено” следующая прививка остается с тем же `scheduledDate`

## Вопросы
1. Что считать “ближайшими” в MVP: следующие `N` по датам или прививки в окне “от сегодня до X дней”?
2. Куда перемещать прививку после отметки “выполнено”: убирать из списка ближайших или показывать отдельно “выполнено”?
3. Разрешено ли “снять отметку” (performed -> planned) в MVP или это запрещено?
4. В каком часовом поясе интерпретируется “сегодня/срок”: на сервере или на стороне клиента?
5. Если в будущем появится несколько питомцев, как выбирается контекст (последний выбранный, переключатель, по умолчанию)?

## Готовность
- Готово к разработке: нет (нужны решения по определению “ближайших” и поведению UI после отметки “выполнено”).
- Что нужно уточнить: пункты в разделе “Вопросы”.
- Следующее действие: зафиксировать правило “ближайшие = ...” и модель отображения выполненных прививок (список vs история).
