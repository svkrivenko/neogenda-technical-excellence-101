# Example Mapping: История 1 — Просмотр списка “хочу посмотреть”

## Story
Как пользователь  
Я хочу видеть список “хочу посмотреть” в одном месте  
Чтобы выбирать, что смотреть дальше, без поиска по заметкам и закладкам

## Rules
1. Rule 1: Отображается актуальный список “хочу посмотреть” (только то, что сейчас находится в watchlist).
   - Пример A: watchlist содержит 3 элемента -> в UI показаны 3 карточки/строки.
   - Пример B (негативный): элемент удалён (или помечен как выведенный из watchlist) -> он не отображается в списке.

2. Rule 2: Если watchlist пустой, показывается понятное состояние “пока нет контента”.
   - Пример A (позитивный/граничный): watchlist = пустой -> отображается сообщение “Список пока пуст” (и, например, кнопка добавить).
   - Пример B (негативный): вместо пустого состояния показывается пустая страница без текста -> это сценарий считается ошибкой UX для истории 1.

3. Rule 3: Для каждого элемента в списке отображается минимальный набор полей, достаточный для выбора.
   - Пример A: есть `title` (название) -> в списке показывается название.
   - Пример B (граничный): отсутствует постер, но есть название -> карточка рендерится без постера, не ломая список.

4. Rule 4: При ошибке загрузки данных список не “падает”; показывается понятное сообщение и возможность повторить.
   - Пример A: при чтении данных произошёл таймаут -> отображается ошибка “Не удалось загрузить список” + опция “Повторить”.
   - Пример B (позитивный): данные успешно загружены -> отображается список.

## Questions
1. Какой минимальный набор полей точно нужен для карточки в MVP: только `title` или ещё тип (фильм/сериал), постер, год?
2. Порядок элементов в списке: по дате добавления, по алфавиту или по последнему изменению — что выбираем для MVP?
3. Как обрабатываем очень большие списки (нужно ли пагинирование/ленивая загрузка уже в MVP)?
4. Watchlist хранится локально (без авторизации/синхронизации) или предполагается серверная синхронизация?
5. Какой текст/формат сообщений об ошибках принимаем для MVP (единые формулировки в UI или допускаем вариативность)?

## Готовность
- Готово к разработке: нет
- Что нужно уточнить: минимальные поля карточки и порядок сортировки для MVP (особенно если есть постер/год/тип).
- Следующее действие: зафиксировать контракт данных watchlist для UI (поля, порядок, состояния: empty/loading/error) и затем согласовать acceptance criteria для истории 1.
