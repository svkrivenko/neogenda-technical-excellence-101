# Architect-review: пользовательские истории и влияние на архитектуру

## 1. Контекст системы, цели и ограничения
**Цель продукта:** трекер фильмов/сериалов с watchlist (“хочу посмотреть”) и историей просмотра, плюс быстрый импорт карточек контента с КиноПоиска.

**Цели MVP:**
1. Добавление контента в watchlist.
2. Отметки “смотрю/посмотрено” и история (как минимум факт: статус + дата).
3. Импорт карточки с КиноПоиска (минимальный набор полей) и отображение в UI.

**Ограничения архитектуры MVP:**
- Внешний источник (КиноПоиск) неизбежно делает интеграцию нестабильной и усложняет тестируемость.
- Избегаем переусложнения: MVP должен оставаться простым, с изоляцией “грязных” деталей парсинга от домена и UI.

## 2. Оценка impact (High/Medium/Low)
**Общий архитектурный impact: Medium/High**, из-за истории 5 (парсинг/импорт КиноПоиска).

- История 1: **Low** (просмотр watchlist и рендер списка по минимальному контракту).
- История 2: **Low** (добавление/валидация элементов watchlist в рамках минимального контракта).
- История 3: **Low** (удаление элементов и связность с историей/статусами).
- История 4: **Low/Medium** (история статусов требует аккуратной модели данных).
- История 5: **Medium/High** (внешняя зависимость, нестабильный источник данных и контракт).

## 3. Архитектурные решения и соответствие практикам (Clean / Hexagonal)
Чтобы снизить риски и повысить поддерживаемость, предлагается следующая декомпозиция:

1. **UI слой**
   - Показывает карточки и состояния загрузки/ошибок.
   - Не знает деталей HTML/парсинга КиноПоиска.

2. **Application/Domain**
   - Содержит доменные сущности/модели: `WatchlistItem` и история статусов.
   - Определяет порт/интерфейс импорта карточек (например, `ImportMovieCardPort`), возвращающий *нормализованные DTO* для UI.

3. **Infrastructure (адаптеры)**
   - Отдельный адаптер `KinoPoiskImporter/Parser`, который:
     - получает внешний источник,
     - извлекает минимальный набор полей,
     - нормализует данные,
     - возвращает DTO.
   - Здесь же реализуются кеширование, ретраи, обработка таймаутов и fallback-логика.

**Tradeoff:** изоляция через порты/адаптеры добавляет небольшой “каркас” в коде, но существенно улучшает тестируемость и снижает связность при изменениях HTML/контракта.

## 4. Технические риски, анти-паттерны и как их гасить

### История 1 (watchlist: просмотр)
1. Риск: рассинхрон данных между состоянием в UI и тем, что хранится/восстанавливается.
   - Снижение: использовать единый минимальный контракт `WatchlistItem` и один источник правды для рендера (домен/приложение).

### История 2 (watchlist: добавление)
1. Риск: некорректная идентификация элементов (дубликаты, разные представления одного фильма/сериала).
   - Снижение: в приложении договориться о минимальном идентификаторе элемента (для MVP — стабильный id там, где возможно; иначе — единый rule для дедупликации по доступным полям).

### История 3 (watchlist: удаление)
1. Риск: удаление элемента ломает связность с последующей логикой статусов/истории (например, если история хранится отдельно).
   - Снижение: определить правила MVP: либо каскадное удаление данных “истории” (если она привязана только к элементу watchlist), либо хранение в виде “архивных” записей без участия UI.

### История 4 (статусы и история)
1. Риск: “история статусов” превращается в сложную систему с редактированием/перепланированием.
   - Снижение: для MVP хранить только события вида `status + timestamp` (без “редактирования прошлого”).
2. Риск: неконсистентные даты/таймзоны при отображении.
   - Снижение: хранить дату в унифицированном формате, а локализацию/форматирование делать на UI.

### История 5 (карточки из КиноПоиска)
1. Риск: нестабильность парсинга (структура страницы меняется).
   - Снижение: парсить только минимальный набор полей; иметь fallback (карточка собирается из того, что удалось извлечь, и не ломает экран).
2. Риск: rate limit / таймауты / “хрупкая” сеть.
   - Снижение: кеш на уровне импортера (TTL), ограниченный retry, явные таймауты; UI должен отображать статус загрузки.
3. Риск: дедупликация и некорректное сопоставление фильмов/сериалов.
   - Снижение: нормализовать ключевой идентификатор импортируемого контента (по возможности единый стабильный id КиноПоиска); опираться на него при добавлении в watchlist.
4. Риск: неполные данные (отсутствует постер/часть текста).
   - Снижение: “graceful degradation” в UI: карточка рендерится с доступными полями, отсутствующие — скрываются/заменяются плейсхолдерами.
5. Риск: производительность (импорт может быть медленным).
   - Снижение: асинхронный импорт; не делать массовый обход; кэшировать результаты.
6. Риск: тестируемость (невозможно надежно тестировать парсинг без зафиксированных данных).
   - Снижение: использовать заготовленные HTML/JSON-фикстуры для контрактных тестов парсера и мокать HTTP в unit-тестах.

## 5. Рекомендации по следующему шагу (next steps) и план валидации
1. Определить **контракт импорта**: какой минимальный набор полей возвращает адаптер (DTO) и как выглядит error/fallback.
2. Реализовать парсер как **изолированный адаптер** и покрыть его:
   - unit-тестами на фикстурах (предсказуемые страницы),
   - контрактными тестами на типовых случаях (фильм/сериал, неполные поля).
3. Добавить в приложение дедупликацию по стабильному id (или по наиболее надежному идентификатору, доступному из импортируемых данных).
4. Валидация MVP:
   - UX: отсутствие падений при сбоях импорта (карточка/ошибка/повтор попытки).
   - Данные: корректность добавления в watchlist и отображение статусов.
   - Надежность: ограничения по таймаутам/ретраям и повторяемость результатов по кешу.
