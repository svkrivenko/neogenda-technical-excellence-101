# Architect-view: Архитектурная оценка пользовательских историй

## Контекст
Продукт: трекер прививок для владельцев котов и собак.

Ожидаемый MVP (по пользовательским историям):
- пользователь добавляет питомца (вид + дату рождения) и получает персональный план обязательных прививок с датами;
- пользователь видит список ближайших запланированных прививок и отмечает “выполнено”;
- поддерживается несколько питомцев в одном аккаунте/приложении.

## История 1: Персональный план после добавления питомца
### Сложность реализации: Medium
Основная работа — корректно вычислить календарные даты относительно `birthDate` и привязать их к набору обязательных шаблонов.

### Технические риски
- Корректность медицинского расписания: риск ошибок в базовых “обязательных” шаблонах и смещениях.
- Семантика дат: часовые пояса/округления могут сместить дату на сутки при вычислениях.
- Валидация входных данных: `birthDate` может быть введена неточно (ошибка пользователя/неизвестность).
- Управляемость правил: при росте количества типов прививок шаблоны могут стать трудно поддерживаемыми.

### Рекомендации для MVP (минимизация риска)
- Начать с узкого поднабора: сначала базовый график **только для котов** и **только обязательные прививки**, без кастомных.
- Держать вычисления в “date-only” подходе (дата без времени) и явно определить правила округления/границы дат.

## История 2: Просмотр ближайших прививок и отметка статуса
### Сложность реализации: Low-Medium
Функциональность сводится к отображению списка предстоящих запланированных прививок и обновлению статуса.

### Технические риски
- Расхождение между “планом” и “статусом”: пользователь может отметить выполненной прививку, у которой уже изменились/пересчитались даты.
- Определение “ближайших”: нужны четкие правила сортировки и окна “предстоящих”.

### Рекомендации для MVP
- Явно хранить “scheduledDate” в записи плановых прививок и отдельно хранить статус (planned/performed).
- Для первой версии ограничить логику “ближайших” простым правилом: сортировка по дате и показ фиксированного ближайшего диапазона.

## История 3: Учет нескольких питомцев
### Сложность реализации: Low
Нужно обеспечить разделение данных по `petId` и удобный способ переключения.

### Технические риски
- Смешение данных между питомцами при выборке/обновлении статуса.

### Рекомендации для MVP
- Всегда фильтровать данные по `petId` при чтении списка прививок и при обновлении статуса.

## Резюме по общим рискам
- Главный риск — корректность и поддерживаемость “расписания обязательных прививок” и вычислений дат.
- Второй риск — дата/время (часовые пояса, округления) и то, как определяется “в срок”.

## Next steps (что делать дальше)

  - добавление собак;
  - затем кастомные прививки (если они нужны по продукту);
  - интеграции (например, Google Calendar) — после базовой ценности.
