# Правила работы

## Общие принципы

1. **Краткость**: Отвечать предельно кратко, без воды.

2. **Поэтапность**: Двигаться по этапам, не переходить к следующему без завершения предыдущего и подтверждения.

3. **Маленькие шаги**: Двигаться маленькими шагами.

4. **Строгость выполнения**: Выполнять СТРОГО только те инструкции, которые явно указаны в текущем запросе/промпте. ИГНОРИРОВАТЬ всю остальную информацию из файла, других секций, примеров или контекста, даже если она видна. Если что-то не упомянуто в текущем запросе - это не нужно делать. Не предполагать, не додумывать, не использовать "логичные" следующие шаги из других частей файла.

5. **Простота**: Не переусложнять решение, использовать принципы KISS, YAGNI и бережливого подхода.

6. **Трекинг**: Трекать этапы и шаги в файле docs/tasklist.md.

7. **Язык**: Отвечать только по-русски.

8. **Честность**: Переспрашивать, если не уверен. Не бояться признаться, что чего-то не знаешь.

9. **Ограничение области**: Не менять никакие файлы, кроме Unit_7.

## Принципы разработки

### KISS (Keep It Simple, Stupid)
Делать решение максимально простым. Избегать сложных архитектур и абстракций, которые не нужны прямо сейчас.

### YAGNI (You Aren't Gonna Need It)
Не добавлять функциональность, которая не требуется в текущий момент. Не предполагать будущие потребности.

### Бережливый подход (Lean)
- Фокусироваться на ценности для пользователя
- Минимизировать отходы (лишний код, неиспользуемые функции)
- Быстро получать обратную связь
- Итеративно улучшать продукт
